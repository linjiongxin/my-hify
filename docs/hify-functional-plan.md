# Hify 功能规划文档

> 基于 Dify 简化版的 AI Agent 平台，面向 20-50 人团队内部使用
>
> 整理时间: 2026-04-09
> 约束条件: 1人开发，本地部署

---

## 一、技术栈

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| 前端 | **Vue 3 + TypeScript + Pinia + Element Plus** | Vue 3 组合式 API + TS 类型安全，Element Plus 提供完整组件库，快速开发 Admin 界面 |
| 后端 | Java + Spring Boot 3.x | 现有项目基础 |
| 数据库 | **PostgreSQL 15+** | 用户、工作流定义、对话记录，pgvector 插件支持向量检索 |
| ORM 框架 | **MyBatis + MyBatis-Plus** | 快速 CRUD、代码生成、分页插件，提升开发效率 |
| 向量库 | **PostgreSQL + pgvector** | 复用主数据库，避免额外运维成本，pgvector 性能足够支撑 20-50 人团队 |
| 缓存/消息 | **Redis** | 缓存 + 流式消息 Pub/Sub |
| MCP | 自定义实现 SSE 传输 | 兼容 Anthropic MCP 协议规范 |
| 部署 | Docker Compose | MySQL + Redis + Hify 一键启动 |

## 二、MCP (Model Context Protocol) 功能规划

### MCP 角色取舍

| MCP 角色 | 优先级 | 说明 |
|---------|--------|------|
| **作为客户端（调用外部 MCP）** | ✅ 必须做 | 让 Hify Agent 能调用外部工具（高德地图、数据库查询等），这是平台核心价值 |
| **作为服务端（暴露为 MCP）** | ❌ 二期再做 | 让 Cherry Studio 等调用 Hify 工作流，内部使用场景需求低，开发成本高 |

### MCP 简化实现范围

```yaml
实现范围:
  传输协议:
    - SSE (Server-Sent Events)  # 只做 SSE
  不做:
    - stdio
    - WebSocket

  能力:
    - Tools      # 工具调用（必须）
    - Resources  # 资源访问（可选）

  不做:
    - Prompts    # 提示词模板（用 Hify 自己的 Prompt 管理替代）
    - Sampling   # 反向采样（复杂，内部用不上）
```

### MCP 工具发现流程

1. 管理员在 Hify 配置页面添加 MCP Server URL
2. Hify 启动时调用 `list_tools` 获取工具列表存入 MySQL
3. Agent 配置时从列表中选择要使用的工具
4. 运行时通过 SSE 调用 `call_tool`

## 三、功能取舍详表

### ✅ 必须做的核心功能（MVP）

| 功能模块 | 具体实现 | 理由 |
|---------|---------|------|
| **模型网关（两级架构）** | **模型提供商管理**：配置各大厂商 API 接入（OpenAI、深度求索、通义千问等），包含基础 URL、认证密钥等<br><br>**模型管理**：每个提供商下挂多个具体模型（如千问下有 2.5/3.5/Max 等），记录模型能力标签（是否支持工具调用、视觉、流式）、上下文窗口、默认参数等 | 两级架构更清晰：先选厂商，再选模型。支持同一厂商多模型切换（如 GPT-4/GPT-4o），统一调用抽象层方便后期扩展 |
| **基础 Agent 能力** | ReAct + Function Call，工具调用抽象层 | 核心卖点，MCP 工具与内置工具对 Agent 无差别 |
| **MCP 客户端** | SSE 传输，动态工具发现 | 扩展工具生态的关键，避免自己维护大量工具 |
| **上下文管理** | Session 级别键值存储，支持多轮参数传递 | MCP 的核心价值，解决"记忆"问题 |
| **简单工作流编排** | 6 个核心节点：LLM、知识检索、代码执行、条件分支、HTTP 请求、输入/输出 | 团队自动化流程的基础需求 |
| **基础 RAG 知识库** | 支持 PDF/Markdown/TXT，固定长度分块策略 | 内部文档问答是高频场景，80% 问题可用简单分块解决 |
| **用户登录与权限** | JWT + 两级角色（管理员/普通用户） | 20-50 人需要区分权限，简单够用 |
| **对话管理** | 会话历史存储、新建对话、删除对话 | Agent 产品的基本体验 |
| **Redis 流式输出** | SSE 推送到 Redis Pub/Sub，前端订阅 | 实时响应体验，避免长轮询 |

### ❌ 明确砍掉的功能

| 功能 | 砍掉理由 |
|------|---------|
| **Chatflow vs Workflow 区分** | 统一用一套工作流引擎，支持开关"记忆"选项即可。减少一套 UI 和逻辑维护成本。 |
| **人机协同 (HITL)** | 实现复杂（需要暂停/恢复状态机），内部使用场景可以先在群里@人确认，人工介入后重新触发。 |
| **Prompt IDE 多模型对比** | 一个人开发，Prompt 版本管理做简单历史记录即可，不需要并排对比功能。 |
| **复杂分块策略** | 只做固定长度分块，不做父子分层/Q&A提取。够解决 80% 场景，降低 RAG 模块复杂度。 |
| **LLMOps 监控大屏** | 用日志文件 + 简单统计页面代替，不需要 Prometheus/Grafana 集成，节省部署成本。 |
| **A/B 测试** | 团队小（20-50人），直接问反馈比数据测试更有效，开发成本高收益低。 |
| **50+ 内置工具** | 只实现搜索、代码执行、HTTP 请求、数据库查询、文件读取 5 个最核心的，其余通过 MCP 扩展。 |
| **企业级 SSO** | 内部部署用账号密码登录即可，不需要对接企业微信/钉钉 SSO，降低认证复杂度。 |
| **审计日志（等保级）** | 记录谁在什么时候调用了什么即可，不需要 6 个月留存、字段级加密。 |
| **多租户隔离** | 整个团队就是一个租户，不需要空间隔离，简化权限模型。 |
| **BaaS API 对外开放** | 内部使用，先不做 REST API，只做 Web 界面，降低接口维护成本。 |
| **问题分类器节点** | 可以用 LLM 节点 + 条件分支实现，不需要单独节点类型，减少节点维护数量。 |
| **模板节点** | 代码节点可以替代，减少一个节点类型降低维护成本。 |

### ⚠️ 可选（后期再加）

| 功能 | 建议 |
|------|------|
| **代码节点沙箱** | 初期直接用 Docker 隔离执行，不做完整的 SSRF 代理网络。 |
| **图片/文件上传** | 先做文本对话，多模态二期再加。 |
| **插件系统** | 内置工具 + MCP 够用，插件架构后期再扩展。 |
| **MCP 服务端** | 等团队有对外暴露能力的需求时再做。 |

## 四、与原始 Dify 功能对比

| Dify 功能 | Hify 决策 | 说明 |
|----------|----------|------|
| 200+ 模型支持 | ❌ 砍到 4-5 个内置厂商 | 保留核心厂商（OpenAI、DeepSeek、通义千问、智谱），每厂2-3个模型 |
| **模型提供商 + 模型两级管理** | ✅ 保留 | 保留两级架构：先选提供商，再选具体模型，结构更清晰 |
| Chatflow/Workflow 双模式 | ❌ 统一为工作流 | 减少概念复杂度 |
| 50+ 内置工具 | ❌ 砍到 5 个 | 其余走 MCP 协议扩展 |
| 复杂 RAG 分块 | ❌ 固定长度分块 | 80/20 法则 |
| LLMOps 监控 | ❌ 简单日志 | 内部使用够用 |
| MCP 客户端 | ✅ 必须做 | 差异化核心功能 |
| MCP 服务端 | ❌ 二期 | 需求低成本高 |
| 企业级 SSO/审计 | ❌ 账号密码 | 内部场景简化 |

## 五、开发优先级（约 10 周）

```
第1阶段（2周）：基础架构
├── Vue 3 项目搭建 + Element Plus 组件库
├── Spring Boot 基础工程 + MySQL 集成
├── Redis 集成（缓存 + Pub/Sub）
└── 用户登录模块（JWT + 两级权限）

第2阶段（3周）：Agent 核心
├── 模型网关（两级架构：提供商管理 + 模型管理）
│   ├── 提供商配置（API 密钥、基础 URL）
│   └── 模型管理（模型列表、能力标签、参数配置）
├── ReAct Agent 实现（推理循环）
├── 5 个内置工具（搜索、代码执行、HTTP、数据库查询、文件读取）
└── 基础对话界面（会话管理、历史记录）

第3阶段（2周）：MCP 集成 ⭐ 核心差异化
├── MCP 客户端 SSE 传输实现
├── 外部 MCP Server 配置管理
├── 工具抽象层（内置工具 vs MCP 工具统一）
└── 上下文管理（Session 级键值存储）

第4阶段（2周）：工作流 + RAG
├── Vue 画布工作流编排（6 个节点）
├── 知识库上传（PDF/Markdown/TXT）
├── 向量存储（Redis Vector / Chroma）
└── 简单 RAG 检索流程

第5阶段（1周）：Polish + 部署
├── 流式输出优化（Redis SSE）
├── Docker Compose 部署脚本
├── 文档完善
└── Bug 修复
```

## 六、风险与应对

| 风险 | 应对策略 |
|------|---------|
| MCP 协议理解成本高 | 先做 SSE 传输，参考开源实现 dify-mcp-server |
| 向量库选型 | Redis Vector 如不可用，备选 Chroma（纯 Python，轻量） |
| 工作流画布复杂 | 用 Vue + @vue-flow/core 库，参考开源实现 |
| 1 人开发进度风险 | 严格控制 MVP 范围，非核心功能一律二期 |
