# Hify 性能瓶颈分析

> 基于 K8s 部署架构（2 副本 + PostgreSQL + Redis + 外部 LLM）
>
> 目标场景：50 人内部团队同时在线

---

## 瓶颈排名（按严重程度）

### 1. 🔴 外部 LLM API 调用延迟

| 项目 | 说明 |
|------|------|
| **瓶颈原因** | API 响应不可控（500ms - 30s+），同步调用长时间占用线程 |
| **触发条件** | 单副本并发 > 20；Provider 响应 > 10s；50 人同时对话 |
| **一期处理** | **必须处理** |

**已设计方案：**
- 统一线程池 + Provider 信号量隔离
- 熔断器（失败率 50% 触发）
- 流式空闲检测（30s 超时）

**待验证：** 30s 空闲检测是否足够敏感

---

### 2. 🟠 PostgreSQL 向量检索性能

| 项目 | 说明 |
|------|------|
| **瓶颈原因** | pgvector 大数据量后性能下降；高维向量（1536 维）计算量大 |
| **触发条件** | 单知识库 > 10,000 分块；并发查询 > 20 QPS |
| **一期处理** | **暂不处理** |

**理由：**
- 一期数据量小（内部文档 < 500 页）
- 已预留索引：`ivfflat (embedding vector_cosine_ops)`
- 二期如瓶颈出现，迁移到 Qdrant/Milvus

---

### 3. 🟡 Tomcat 线程池 + SSE 长连接

| 项目 | 说明 |
|------|------|
| **瓶颈原因** | SSE 长连接占用会话对象；LLM 调用需额外线程 |
| **触发条件** | 活跃 SSE > 100；LLM 响应 > 60s；HPA 扩容滞后 |
| **一期处理** | **建议处理** |

**优化措施：**
- HPA 阈值 70% → **50%**（扩容更灵敏）
- 就绪探针敏感度调低，快速剔除异常副本

---

### 4. 🟡 JVM 内存与 GC

| 项目 | 说明 |
|------|------|
| **瓶颈原因** | 堆内存限制 512m；流式缓存 + RAG 结果集大对象 |
| **触发条件** | 上下文 > 100k tokens；并发流式 > 10 个；Full GC > 1次/分钟 |
| **一期处理** | **建议处理** |

**监控指标：**
- 监控 `jvm_memory_used_bytes`
- 若容器 OOM，调整为 **768m**（配合容器 limit 1Gi）

---

### 5. 🟢 Redis 内存与连接

| 项目 | 说明 |
|------|------|
| **瓶颈原因** | 配置 384mb，`allkeys-lru` 淘汰策略 |
| **触发条件** | 会话数据堆积；流式消息积压；内存 > 80% |
| **一期处理** | **暂不处理** |

**理由：**
- 50 人会话数据 < 50MB
- 流式消息即时消费，不会积压

---

### 6. 🟢 Ingress/Nginx

| 项目 | 说明 |
|------|------|
| **瓶颈原因** | SSE 长连接占用连接数；SSL 加解密 CPU |
| **触发条件** | 并发连接 > 8192；SSL 握手风暴 |
| **一期处理** | **无需处理** |

---

## 总结表

| 排名 | 瓶颈 | 严重程度 | 触发条件 | 一期处理 |
|:---:|------|:---:|:---|:---:|
| 1 | **外部 LLM API** | 🔴 高 | 并发 > 20，响应 > 10s | **必须** |
| 2 | **PostgreSQL 向量检索** | 🟠 中 | 文档 > 500 页 | 暂不 |
| 3 | **Tomcat 线程 + SSE** | 🟡 中 | SSE > 100 | 建议 |
| 4 | **JVM 内存/GC** | 🟡 中 | 上下文 > 100k tokens | 建议 |
| 5 | Redis 内存 | 🟢 低 | 内存 > 80% | 暂不 |
| 6 | Ingress/Nginx | 🟢 低 | 连接 > 8192 | 无需 |

---

## 一期必须关注的监控指标

### P0 告警（必须处理）

```yaml
llm_request_duration_seconds{quantile="0.99"} > 30
llm_circuitbreaker_state == 1              # 熔断器 OPEN
jvm_memory_used_bytes / jvm_memory_max_bytes > 0.9
container_memory_working_set_bytes > 700Mi
```

### P1 观察（趋势预警）

```yaml
tomcat_connections_active > 800
postgresql_connections_active > 8
llm_semaphore_waiting > 5                  # 信号量等待队列
```
