审查代码并提供反馈。只审查，不修改代码。

目标: $ARGUMENTS (若未指定则审查当前 git diff)

## 步骤

1. **确定审查范围**
   - 若提供了文件路径，读取该文件及同一包/模块内的关联文件（如 Controller 对应的 Service 接口）
   - 若未提供路径，运行 `git diff` 查看当前变更

2. **读取项目规范**
   - 读取 `CLAUDE.md` 中「代码组织」、「各层职责」、「跨模块调用规则」、「代码审查 Checklist」相关章节
   - 若审查涉及数据库实体/SQL，也读取 `docs/database-guidelines.md`

3. **执行审查**（按文件类型应用对应检查项）

### 通用审查
- **Bug 与错误**: 逻辑错误、空值检查、边界情况、异常处理
- **安全性**: 输入验证、注入风险、敏感数据暴露
- **性能**: 明显的低效问题、N+1 查询、内存泄漏、不必要的对象创建
- **可读性**: 命名规范、复杂度、文档需求

### 架构与项目规范审查（关键）
根据被审代码所在包路径和文件类型，重点检查：

**若文件在 `api/` 包中：**
- [ ] 是否定义了接口和 DTO？
- [ ] 接口返回值是否为 DTO/VO，禁止返回 Entity？
- [ ] 是否使用了内部类或非标准包中的类型？

**若文件在 `controller/` 包中：**
- [ ] 是否只负责「接收参数、入参校验、权限检查、调用、返回」？
- [ ] 是否包含业务逻辑（如字段拷贝、条件组装、查库后转换）？
- [ ] 返回值是否为 DTO/VO，禁止直接返回 Entity？
- [ ] 是否直接注入并调用了其他模块的 Service/Mapper？

**若文件在 `service/` 包中：**
- [ ] 实现类是否加了 `@Transactional`（类级别或方法级别）？
- [ ] 读操作是否加了 `readOnly = true`？
- [ ] 是否直接调用了其他模块的 Service/Mapper（应通过 `api/` 层）？
- [ ] 是否存在与其他模块的双向注入（循环依赖）？

**若文件在 `mapper/` 包中：**
- [ ] 复杂 SQL 是否写在 XML 中？
- [ ] 是否存在 `@Select`、`@Update`、`@Delete`、`@Insert` 等注解 SQL？
- [ ] Mapper 类型是否被其他模块引用？

**若文件在 `entity/` 包中：**
- [ ] Entity 是否继承了 `BaseEntity`？
- [ ] 是否包含业务逻辑方法？
- [ ] 表名前缀是否符合模块命名（如 `model_`、`agent_`、`chat_`）？

**跨模块调用（全局）：**
- [ ] 其他模块若要调用本模块能力，是否提供了 `api/` 层接口？
- [ ] 是否存在直接注入其他模块 Service/Mapper 的情况？

## 输出格式

```markdown
## 代码审查

### 概述
[一句话总体评价]

### 发现的问题

#### 架构/规范（必须修复）
- [问题]: [位置] - [简要说明]

#### 严重（必须修复）
- [问题]: [位置] - [简要说明]

#### 警告（建议修复）
- [问题]: [位置] - [简要说明]

#### 建议（可选优化）
- [建议]: [位置] - [简要说明]

### 优点
- [正面观察]
```

## 指南
- 架构规范问题必须放在「架构/规范」分类下，优先级最高
- 明确指出位置（尽可能精确到 `file:line`）
- 提供可操作的反馈，说明应该如何整改（但**不要修改代码**）
- 除非影响可读性，否则不要纠结于代码风格
- 认可看到的优秀实践
- 若某条规范不适用当前文件类型，无需强行列出
